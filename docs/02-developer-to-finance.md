# 程序员转金融领域技能图谱

最后更新：2026-04-30

程序员转 FinTech 的优势是工程能力、抽象能力和数据处理能力。需要补齐的是金融业务语言、监管意识、数据质量意识和风险思维。

## 已有能力如何迁移

### 后端开发

可迁移到支付、账务、风控、银行 API、交易系统、合规报送。需要重点补充金额精度、事务一致性、幂等、审计日志、权限和状态机。

### 数据分析

可迁移到交易流水分析、信贷风控、投资组合分析、反欺诈、经营报表。需要重点补充金融指标、数据血缘、异常值处理和模型解释。

### 前端开发

可迁移到银行后台、交易终端、投资分析工具、风控工作台。需要重点补充表格、筛选、权限、操作确认、错误提示和审计体验。

### DevOps 和安全

可迁移到金融系统稳定性、安全合规、日志监控、密钥管理、灾备。需要重点补充监管要求、数据保护、访问控制和变更审计。

## 必补知识

### 金融基础

- 资产、负债、权益
- 收入、支出、现金流
- 利率、复利、贴现
- 收益率、波动率、最大回撤
- 股票、债券、基金、衍生品的基本区别
- 银行、券商、支付机构、清算机构分别做什么

### FinTech 业务

- 支付订单、退款、拒付、对账
- 银行账户、交易流水、开放银行
- 信贷申请、授信、还款、逾期
- KYC、AML、制裁筛查、可疑交易监控
- 投资组合、风险评估、适当性

### 工程能力

- 金额类型和精度处理
- 数据库事务和一致性
- 幂等键和重复请求处理
- 状态机设计
- 审计日志和事件溯源
- API 安全、认证和授权
- 批处理、报表、对账

### 数据和模型

- SQL 聚合和窗口函数
- Pandas 数据清洗和分析
- 基础统计：均值、方差、相关性、分布
- 风险指标：波动率、回撤、违约率
- 机器学习基础：分类、回归、过拟合、特征工程、模型监控

### 合规和风险意识

- 不同地区监管规则不同，不能把一个市场的规则直接套到另一个市场。
- 监管、法律、税务、会计内容必须查证官方或专业来源。
- 学习项目可以模拟 KYC、AML、风控，但不能把模拟规则当作真实合规意见。
- 金融产品解释要避免变成个性化投资建议。

## 推荐学习顺序

1. FinTech 全景和金融基础词汇。
2. 双分录账本：账户、交易、余额、审计。
3. SQL 和 Pandas：交易流水分析。
4. 支付系统：订单、退款、回调、幂等。
5. 投资分析：收益率、波动率、组合和回撤。
6. 风控系统：规则、评分、异常检测。
7. 合规基础：KYC、AML、权限、日志、报送。

## 作品集方向

- 双分录账本：展示金融数据正确性和审计能力。
- 支付订单沙盒：展示状态机、幂等、回调和对账。
- 交易流水分析器：展示 SQL、Pandas 和金融报表能力。
- 投资组合实验室：展示收益风险分析能力。
- 风控规则引擎：展示规则设计、异常检测和可解释性。

## 当前最适合的第一个项目

从 `labs/ledger-basics/` 开始，先做一个最小账本：

- 创建账户。
- 录入交易。
- 每笔交易至少包含两条分录。
- 借方金额合计必须等于贷方金额合计。
- 可以查询账户余额。
- 每次交易都保留审计记录。

这个项目规模小，但覆盖了 FinTech 系统最核心的工程思维。
