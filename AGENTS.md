# AGENTS.md

本文件面向后续进入本仓库的 AI 终端。目标是保证协作学习能够连续、可验证、面向实践，而不是生成没有出处的金融概念。

## 协作对象

- 用户是程序员，金融和 FinTech 按零基础处理。
- 默认使用中文沟通；必要时保留金融术语的英文原文，例如 account、transaction、settlement、liquidity。
- 解释应循序渐进，避免一次性堆叠大量术语。
- 学习目标是“理解金融科技并能做出工程作品”，不是短期投资建议。

## 语言和编码约定

- 文档类文件，尤其是学习资料，使用中文编写，并按 UTF-8 编码阅读和保存。
- 脚本、命令行输出、测试样例中的用户可见文本、错误示例文本，以及未来前端展示给用户的 UI 文案，默认使用英文。
- 脚本注释可以使用中文，尤其是在解释金融概念或学习意图时。
- 代码标识符、文件名、API 字段名和配置项使用英文。

## 每次开始前必须读取

1. `LEARNING_PROGRESS.md`：确认当前学习进度、近期计划和交接记录。
2. `README.md`：确认仓库结构和学习方式。
3. 与当前任务相关的 `docs/` 或 `labs/` 文件。

## 资料真实性要求

- 不得编造金融知识点、监管要求、API 行为、市场数据、认证规则或资料来源。
- 区分两类内容：
  - 稳定基础知识：例如账户、现金流、收益率、借贷记账、风险、清算、结算。
  - 时效性知识：例如监管政策、API 文档、产品价格、考试认证、市场数据、法律法规、行业报告。
- 对时效性知识，必须先查证官方或专业来源，并在回答或文档中注明来源。
- 如果无法确认，应明确写成“待查证”，不能把推测写成事实。
- 引用资料时优先使用一手来源；博客、论坛、社媒和未署名文章只能作为辅助线索，不能作为主要依据。

## 推荐权威来源

优先使用 `docs/00-authoritative-sources.md` 中维护的来源。常用类别包括：

- 国际组织：BIS、World Bank、IMF、FSB、IOSCO。
- 专业机构：CFA Institute、GARP。
- 美国监管和公共机构：Federal Reserve、SEC、FINRA、CFPB、OCC、FDIC。
- 中国监管和公共机构：PBOC、中国证监会、国家金融监督管理总局、国家外汇管理局。
- 官方开发者文档：Visa Developer、PayPal Developer、Plaid Docs、Open Banking UK、Stripe Docs。

## 教学方式

讲解一个知识点时，优先使用这个结构：

1. 中文定义。
2. 英文术语。
3. 为什么金融系统需要它。
4. 程序员实现时会遇到什么问题。
5. 一个最小例子，能写代码时尽量落到 `labs/`。

对数学和金融公式，先解释直觉，再给公式，再用小数据演示。不要默认用户已经理解金融背景。

## 工程方式

- 初期技术栈优先使用 Python、SQLite、pytest、Pandas。
- Python 环境优先使用 Anaconda / conda 管理；如需要新增依赖，优先更新 `environment.yml`。
- 不要过早引入复杂框架；只有实验需要 API 服务时再考虑 FastAPI。
- 每个实验应尽量可运行、可测试、可解释。
- 金融系统相关代码要强调正确性、幂等性、审计、权限和数据一致性。
- 保持改动范围小，避免无关重构。

## 进度维护

完成以下任一事项后，更新 `LEARNING_PROGRESS.md`：

- 新增或完成一个学习主题。
- 新增或完成一个代码实验。
- 调整未来计划。
- 发现重要资料来源。
- 形成新的关键结论或待查证问题。

更新时至少记录：

- 日期。
- 做了什么。
- 下一步是什么。
- 是否存在待查证内容。

## 边界

- 不提供个性化投资建议。
- 不替代法律、税务、会计、合规或持牌金融顾问意见。
- 示例中的金融数据仅用于学习，不代表真实市场判断。
